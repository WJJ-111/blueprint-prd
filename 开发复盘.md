# VibeCoding 开发复盘

> 记录时间：2026-04-05

---

## 一、生产过程中的问题与解决

### 1. pip 安装超时 + 版本冲突

**问题**：`pydantic-core==2.14.1` 找不到 + PyPI 连接超时。根因有两个：一是 `requirements.txt` 用了 `==` 精确锁版本，国内网络拉不到对应 wheel 文件；二是 pip 版本太旧（21.2.4）解析依赖能力弱。

**解决**：`==` 改为 `>=`，`start.sh` 先升级 pip，同时切换清华/淘宝镜像。

---

### 2. uvicorn 命令找不到

**问题**：`start.sh` 直接调 `uvicorn`，但通过 `pip3 install --user` 安装的包，可执行文件在 `~/Library/Python/3.x/bin/` 下，不一定在 PATH 里。

**解决**：改用 `python3 -m uvicorn`，跟随当前 Python 环境走，更可靠。

---

### 3. 前端 ECONNREFUSED，但后端其实没起来

**问题**：`start.sh` 里 sleep 2 秒不够，后端还在安装依赖时前端已经启动并开始请求，日志看起来像后端挂了，实际是启动慢。

**解决**：sleep 改为 3 秒。根本上应改为轮询健康检查（`/health`）而不是硬等。

---

### 4. 401 无效令牌，排查绕了弯

**问题**：用户填好 `.env` 后重启服务，但端口 8000 被旧进程占用，新进程起不来，旧进程带的是空 API Key，所以一直 401。排查时走了"langchain 参数传错"的弯路，直到单独用 curl 测通了才确定是进程问题。

**解决**：`lsof -ti :8000 | xargs kill -9` 清掉旧进程。根本上 `start.sh` 应该在启动前主动检查并清理端口。

## 二、初始提示词应补充的内容

回头看，以下几点如果在提示词里明确，能省掉很多来回：

| 遗漏点 | 影响 |
|--------|------|
| **流式输出（streaming）** | AI 回复较长时用户需要等全部返回才能看到，体验差，应默认 streaming |
| **"生成 PRD"按钮的出现时机** | 模糊导致实现了"一直显示但 disable"的方案，后来才改 |
| **PRD 输出方式**（下载 / 预览 / 复制） | 第一版实现了子页面预览，后来又改成下载，多做了工作 |
| **API Key 配置入口在前端还是后端** | 实现了前端配置面板，后来又删，方向反复 |
| **部署环境**（本地单人使用 vs. 多人） | 影响是否需要会话隔离、端口安全等设计决策 |
| **是否需要 PRD 迭代修改** | 实现了 refine 功能又废弃，如果明确就不用写 |

## 三、对工具本身的建议

### 体验层

- **流式输出**：AI 回复现在是等全部生成才显示，用户体验割裂。应改为 SSE/WebSocket 流式返回，边生成边显示。
- **PRD 生成前预览确认**：下载前弹出预览窗口，用户可以快速审阅再决定是否下载或直接复制给 Claude Code。
- **会话丢失问题**：刷新页面对话就没了。可以用 `localStorage` 存最近一次会话（消息列表 + 表单数据），无需后端持久化。

### 功能层

- **对话中途修改表单**：目前填完表单就锁死了，应允许在对话中回头补充/修改，AI 能感知到变更。
- **PRD 分段重新生成**：用户可能只想改"技术规格"部分，不必重跑全流程。
- **多份 PRD 对比**：同一产品想法，用不同侧重生成两版（to VC 版 vs. to 工程师版）。
- **导出格式扩展**：目前只有 `.md`，可加 `.txt`（纯文本）供非技术用户。

### 提示词层

- 当前系统提示词没有明确要求 AI **在对话中途主动推进**（比如"我觉得你的 MVP 功能已经够清晰了，要不要现在生成 PRD？"），AI 往往只被动回答，需要用户自己判断何时结束。
- PRD 模板中的"非功能需求"和"项目范围"两节经常被 AI 写得很空洞，提示词应给出更具体的填写要求。

## 四、表单字段扩展建议

### 基础问题（所有人适用）

| 建议新增字段 | 原因 |
|-------------|------|
| **竞品参考** | "类似 Notion 但更轻量"这类描述能让 AI 快速定位产品定位，节省大量对话轮次 |
| **核心使用场景（1-2 个）** | 用户故事维度，比"目标用户"更能帮 AI 理解功能优先级 |
| **最不能接受的点** | 反向约束，如"绝对不能有广告""不能需要注册才能用"，往往决定架构 |

### 高级问题（开发者）

| 建议新增字段 | 原因 |
|-------------|------|
| **现有系统集成** | 是否要对接已有的 ERP/CRM/内部系统，直接影响技术方案 |
| **数据敏感级别** | 涉及个人隐私、医疗、金融数据需要特殊处理，PRD 里要体现 |
| **团队规模 & 技术背景** | 2 人团队和 20 人团队的技术选型完全不同 |
| **上线时间压力** | 影响 MVP 的取舍逻辑，"两周上线"和"三个月上线"的 PRD 应该完全不一样 |
